Use the repository's written documentation as review context and check whether
the change matches the documented intent.

- Start from README.md, ARCHITECTURE.md, AGENTS.md, and CONTRIBUTING.md for
  architectural context.
- Check whether code, tests, and documentation all tell the same story. Flag
  mismatches between implementation and the documented architecture.
- Treat AGENTS.md as the authoritative guide for sharp edges and invariants.
  If a change violates an invariant documented there (for example, calling
  getType().getProperties() on a DOM wrapper type), flag it.
- If CI or another required check already enforces a merge rule, do not ask
  for duplicate PR template sections or manual checklists.
- For any change to the React extractor
  (packages/experience-design-system-cli/src/analyze/extract/react.ts), check
  whether AGENTS.md's "DOM prop inflation" guidance was considered. If the
  change adds a new extraction path that could inflate prop counts, flag it.
- Distinguish the current shipped API surface (RawComponentDefinition, CDFFile,
  DesignToken types in @contentful/experience-design-system-types) from
  internal implementation details. Public type changes require extra scrutiny
  because they ship to consumers via npm.
